# 宇树 G1 室内导航技术路线（FAST-LIVO2 建图 + AMCL 定位 + 2D 地图导航）

## 1. 目标定义

当前目标不是一步做到“完整高难度足式自主导航”，而是先建立一条**稳定、可调试、容易落地**的室内导航链路，使宇树 G1 在平地室内环境中具备以下能力：

- 使用 Mid-360 + IMU 完成环境建图
- 基于已有地图完成机器人 2D 定位
- 在 2D 地图中进行路径规划
- 能够从起点自主移动到目标点
- 具备基础避障能力

本阶段默认将 G1 **近似看作一个平面移动平台**，即主要关注：

- `x`
- `y`
- `yaw`

暂时不处理楼梯、跨障、复杂地形等高阶足式导航问题。

---

## 2. 总体技术路线

整体方案确定为：

**FAST-LIVO2 建图 + 2D 地图生成 + AMCL 定位 + 2D 导航规划 + G1 运动控制**

其逻辑关系如下：

1. 使用 **FAST-LIVO2** 获取 Mid-360 + IMU 的实时位姿与 3D 点云地图
2. 将 3D 点云地图转换为导航可用的 **2D 栅格地图**
3. 将 Mid-360 实时点云转换为 **2D LaserScan 或等效 2D 障碍观测**
4. 使用 **AMCL** 在 2D 地图上完成机器人定位
5. 使用 2D 导航栈完成：
   - 全局路径规划
   - 局部避障
   - 速度控制输出
6. 将导航输出速度命令转换为 G1 可执行的运动控制命令

---

## 3. 系统分层理解

为了便于理解，可以将整套系统分成五层。

---

### 3.1 传感器层

负责提供原始感知数据。

当前硬件输入包括：

- Mid-360 雷达点云
- Mid-360 内部 IMU
- 宇树机器人本体状态（后续导航控制时可能还会用到）

这一层的任务是：

- 正确接收 DDS 数据
- 桥接到 ROS1
- 处理时间戳、坐标系、TF 关系
- 向上层提供稳定的 ROS 话题

当前你已经基本完成这一层。

---

### 3.2 建图与里程计层

这一层由 **FAST-LIVO2** 负责。

它的主要作用不是“完整导航”，而是：

- 融合激光与 IMU
- 输出实时位姿估计
- 生成 3D 点云地图
- 提供较连续、较平滑的运动轨迹

这一层解决的问题是：

> 机器人在运动过程中，如何持续估计“我相对环境的位置和姿态”。

需要明确的是：

- FAST-LIVO2 更偏向于 **建图 / 里程计 / 前端定位**
- 它不是完整导航系统
- 后续导航仍需额外接入定位与规划模块

---

### 3.3 地图层

导航不直接使用原始 3D PCD 地图，而需要转换为更适合规划和定位的数据形式。

本方案中，地图层包含两类地图：

#### 1）3D 点云地图

来源于 FAST-LIVO2。

作用：

- 作为原始建图结果保存
- 供后续生成 2D 地图
- 供调试、可视化、分析误差使用

#### 2）2D 栅格地图

由 3D 点云投影得到。

作用：

- 供 AMCL 做 2D 定位
- 供 2D 导航栈做全局规划
- 供局部代价地图构建障碍物信息

因此你后续导航真正使用的核心地图，不是 `.pcd`，而是：

- `.pgm`
- `.yaml`

或者等价的 occupancy grid。

---

### 3.4 定位层

这一层选用 **AMCL**。

AMCL 的定位目标是：

> 在一张已知 2D 地图中，根据实时 2D 激光观测，估计机器人当前位姿。

它的典型输入包括：

- 已知 2D 栅格地图
- 实时 `LaserScan`
- 机器人运动里程信息（通常表现为 `odom -> base_link`）

它的输出是：

- `map -> odom`

也就是说，AMCL 的作用不是代替 FAST-LIVO2 建图，而是：

- 在已有地图中重新定位
- 持续修正机器人在地图中的位置
- 为导航系统提供全局一致的位置参考

之所以选 AMCL，而不选更复杂的 3D 点云定位方案，主要原因是：

- 集成成熟
- 稳定性高
- 资料多
- 参数相对容易理解
- 与 2D 导航栈兼容最好
- 更适合小白入门

---

### 3.5 规划与控制层

这一层是“真正让机器人走起来”的部分。

它通常包含两个子模块。

#### 1）全局规划

作用：

- 根据当前位姿和目标点
- 在 2D 地图中规划一条从起点到终点的路径

输出：

- 一条全局路径

#### 2）局部规划 / 局部控制

作用：

- 根据实时障碍物情况
- 在局部范围内调整运动方向和速度
- 避开人、椅子、临时障碍等动态物体

输出：

- 速度命令，如 `cmd_vel`

然后再将这些速度命令适配到 G1 的运动控制接口。

## 4. 为什么选择这条路线

## 4.1 为什么建图继续用 FAST-LIVO2

因为你已经基本打通了：

- Mid-360 数据接入
- ROS1 bridge
- TF 坐标系
- FAST-LIVO2 建图链路

继续保留 FAST-LIVO2 的好处是：

- 不需要重新换一整套 3D SLAM 框架
- 能继续利用已有工作成果
- 对 Mid-360 + IMU 的适配已经有基础
- 3D 建图质量相对较好

因此建图阶段继续用 FAST-LIVO2 是合理的。

---

## 4.2 为什么定位选择 AMCL

AMCL 适合当前阶段，原因有：

- 它专门解决“已有地图中的 2D 定位”
- 目标单一，不和建图功能耦合
- 调试路径清晰
- 社区资料丰富
- 成功率高
- 和 2D 导航栈天然兼容

对你当前这种“先把导航跑通”的需求来说，AMCL 是更稳妥的第一选择。

---

## 4.3 为什么优先做 2D 导航

因为 2D 导航是最成熟、最容易落地的工程路线。

虽然你使用的是 3D 雷达，但当前目标是室内平地移动，因此没有必要一开始就做完整 3D 导航。

2D 导航的优点是：

- 系统复杂度低
- 调试工具成熟
- 可视化直观
- 与 AMCL、全局规划、局部规划兼容性最好

因此现阶段最合适的策略是：

**3D 建图，2D 定位，2D 规划。**

---

## 5. 完整数据流

整个系统的数据流可以概括为：

### 建图阶段

`Mid-360 点云 + IMU`
→ `ROS1 bridge`
→ `FAST-LIVO2`
→ `实时位姿 + 3D 点云地图`

### 地图准备阶段

`3D 点云地图`
→ `高度裁剪 / 投影 / 栅格化`
→ `2D 地图（pgm + yaml）`

### 定位阶段

`实时 Mid-360 点云`
→ `2D LaserScan / 2D 障碍观测`
→ `AMCL`
→ `map -> odom`

### 导航阶段

`目标点`
→ `全局规划器`
→ `局部规划器`
→ `cmd_vel`
→ `G1 控制接口`

---

## 6. 后续需要补齐的关键模块

虽然大方向已经明确，但当前系统还缺几个关键环节。

---

### 6.1 3D 点云地图转 2D 地图

你现在有的是 3D 点云地图，但 AMCL 和 2D 导航用的是 2D 栅格地图。

因此需要增加一个中间处理步骤：

- 对点云按高度范围裁剪
- 去除地面、天花板等无关点
- 投影到平面
- 生成占用栅格地图

这是后续导航的核心前置步骤之一。

---

### 6.2 实时点云转 2D 激光观测

AMCL 通常需要输入 `LaserScan`。

但你当前的传感器是 Mid-360 3D 雷达，因此需要把实时点云转换为 2D 扫描数据，或者构造成导航能接受的 2D 障碍观测。

常见做法包括：

- 从某一高度切一层点云生成 2D 扫描
- 将一定高度范围内的点投影成 2D 障碍
- 供 AMCL 和 local costmap 使用

---

### 6.3 G1 控制接口适配

导航栈最终输出的通常是线速度和角速度命令，但 G1 不是普通轮式底盘，因此还需要一个控制适配层。

这个模块的作用是：

- 接收导航系统的速度指令
- 转换成 G1 可执行的运动命令
- 保证机器人稳定行走

当前阶段可以先将 G1 近似看作一个平面移动平台，只做平地慢速导航。

---

## 7. 本阶段建议的实施顺序

现阶段不宜一次性把所有模块一起上齐，建议按阶段推进。

### 阶段 1：稳定建图
目标：

- FAST-LIVO2 长时间稳定运行
- 地图无明显辐射纹、上下颠倒、严重重影
- 导出干净的 3D 点云地图

### 阶段 2：生成可用 2D 地图
目标：

- 从 3D 点云中生成 2D 栅格地图
- 在 RViz 中正确显示
- 地图边界、墙体、障碍物清晰

### 阶段 3：准备 2D 激光输入
目标：

- 将 Mid-360 实时点云转换为 2D LaserScan
- 能在 RViz 中看到稳定的 2D 激光扇面或轮廓

### 阶段 4：接入 AMCL
目标：

- 加载 2D 地图
- 使用实时 2D 激光进行定位
- 在 RViz 中看到机器人位姿稳定收敛
- 能进行初始位姿设置

### 阶段 5：接入导航栈
目标：

- 设置目标点
- 自动生成路径
- 输出速度命令
- 机器人能自主行走到目标点

### 阶段 6：优化避障与控制
目标：

- 提高避障效果
- 减少局部振荡
- 提高路径平滑性与稳定性

---

## 8. 本方案的核心优势

本技术路线的优势在于：

- 尽量复用你已经跑通的 FAST-LIVO2 建图能力
- 将复杂问题分解为“建图、定位、规划、控制”四个清晰模块
- 优先采用成熟稳定的 2D AMCL 定位方案
- 降低入门难度
- 有利于快速得到第一个可运行结果

换句话说，这条路线不是理论上最前沿的，但非常适合当前阶段：

> 先跑通，再优化；先稳定，再高级。

---

## 9. 当前阶段的最终结论

对于宇树 G1 + Mid-360 的室内导航，当前最合适的第一阶段技术路线是：

**FAST-LIVO2 负责 3D 建图与位姿估计；**
**3D 点云地图转换为 2D 栅格地图；**
**实时 3D 点云转换为 2D 激光观测；**
**AMCL 负责 2D 地图定位；**
**2D 导航栈负责路径规划与避障；**
**最终控制 G1 在平地室内环境中自主移动。**

这是一条：

- 稳定
- 成熟
- 可逐步调试
- 对小白相对友好

的技术路线。

---

## 10. 后续工作建议

在这条路线下，接下来最优先的工作不是立即调导航参数，而是先完成两个基础中间件：

1. **3D 点云地图转 2D 栅格地图**
2. **实时 Mid-360 点云转 2D LaserScan**

只要这两个桥接模块打通，AMCL 和 2D 导航就有了落地基础。
